# Specification: Interactive Web-based Explorer

## 1. Objective
To provide a high-fidelity, interactive, and navigable web-based interface for exploring the semantic call graphs and module dependency maps generated by JS Cartographer. This tool aims to replace the static CLI-based visualizations with a dynamic environment that bridges the gap between high-level architectural overview and low-level source code inspection.

## 2. User Personas
- **Security Researchers:** Analyzing obfuscated bundles for potential vulnerabilities or backdoors.
- **Reverse Engineers:** Deciphering complex, minified business logic to understand proprietary algorithms or API interactions.
- **Frontend Developers:** Navigating large, legacy, or undocumented codebases to identify dependencies and call chains.

## 3. Functional Requirements
### 3.1 Codebase Visualization (The Map)
- **Interactive Call Graph:** Nodes represent named functions, edges represent call sites (internal or cross-file).
- **Module Dependency Graph:** Nodes represent files, edges represent import/export relationships.
- **Layout Engine:** Automatic, hierarchical layout with support for manual node movement.
- **Filtering & Search:** Find functions/files by name; filter by depth from a specific entry point.

### 3.2 Source Code Inspection (The Territory)
- **High-Quality Code Viewer:** Syntax highlighting, line numbers, and search within a file.
- **Bidirectional Navigation:**
    - Click a node in the graph -> editor jumps to the file/line where that function/file is defined.
    - Click a function in the code -> highlight the corresponding node in the graph and its neighbors.

### 3.3 CLI-to-Web Bridge
- **Local Server:** A lightweight, embedded HTTP server in the CLI that serves the frontend and project data.
- **Auto-Discovery:** Automatically opens the default browser to the local explorer URL.
- **File Streaming:** On-demand serving of deobfuscated source files from the output directory.

## 4. Non-Functional Requirements
- **Performance:** Handle graphs with 1000+ nodes and 5000+ edges with minimal lag.
- **Reliability:** The explorer must be robust against corrupted or missing graph JSON files.
- **Developer Experience (DX):** Zero-configuration setup (it "just works" after a `humanify` run).
- **Security:** The embedded server must only bind to `localhost` to prevent accidental remote access to local source code.

## 5. Architecture Overview
### 5.1 Technology Stack
- **Frontend:** React, Vite, Tailwind CSS, Zustand.
- **Graph Engine:** React Flow (for its high customizability and interactivity).
- **Code Viewer:** Monaco Editor (the industry standard for web-based code viewing).
- **Backend:** Express.js (embedded within the humanify binary).

### 5.2 Data Flow
1. `humanify <provider> <input>` generates `output/call-graph.json` and `output/module-graph.json`.
2. `humanify explore output/` starts an Express server.
3. The Express server serves a pre-built React frontend.
4. The React frontend fetches graph data from local API endpoints.
5. React Flow transforms the JSON into interactive nodes/edges.
6. Monaco Editor displays source files requested via a dedicated file-fetcher API.

## 6. Security & Safety
- **Local-Only:** The server will listen on `127.0.0.1` by default.
- **Read-Only:** For the MVP, the server will only provide read access to the `output/` directory. No write/delete operations will be exposed.
- **Path Sanitization:** API endpoints must rigorously validate file paths to prevent directory traversal attacks (e.g., `../../etc/passwd`).

## 7. Testing Strategy
- **Frontend Unit Tests:** Using Vitest for graph-to-node transformation logic.
- **Component Tests:** Using React Testing Library for the sidebar, editor, and graph interactions.
- **Integration Tests:** Verifying the Express server's API endpoints (JSON delivery, file streaming).
- **End-to-End (E2E) Tests:** Verifying the full "explore" workflow from command launch to browser rendering using the `webpack-hello-world` fixture.
