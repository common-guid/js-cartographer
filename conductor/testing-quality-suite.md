# Implementation Plan: Quality Evaluation Suite

## Background & Motivation
Currently, JS Cartographer's test suite ensures the application correctly processes inputs without erroring, but it does not measure the *quality* of the unminified and renamed output. Because the tool relies on non-deterministic LLMs to suggest names, we need a way to track whether changes to the prompts, AST traversal, or heuristics improve or degrade the output's readability and correctness over time.

## Proposed Solution
We will introduce an **on-demand quality evaluation suite** that operates independently of the main unit/e2e testing suite. It will utilize two complementary approaches:

1. **Source Map Recovery Score:** An objective metric that compares the names generated by JS Cartographer against the original names embedded in a source map. This yields a hard percentage of recovered context.
2. **Interactive Snapshot Diffing:** A subjective, visual diff tool that compares the newly generated code against known "golden" snapshots. When changes occur (e.g., due to LLM non-determinism or prompt tweaks), the developer is prompted to visually review the diff and decide whether to accept the new output as the new baseline.

## Implementation Steps

### 1. Update Test Fixture
*   Modify `fixtures/webpack-hello-world/package.json` to add a `build:sourcemap` script.
*   Add a secondary Webpack config (or use environment variables) to build the fixture with `devtool: 'source-map'`, producing `bundle.js.map` alongside `bundle.js`.
*   Ensure the existing `dist/bundle.js` used by E2E tests remains untouched and unaffected by this secondary build.

### 2. Install Required Dependencies
*   Add `source-map` for parsing source maps.
*   Add `diff` or a similar library for generating human-readable diffs in the terminal.
*   Add `@inquirer/prompts` or use Node's native `readline` for the interactive "Accept snapshot?" prompt.

### 3. Implement Source Map Recovery Scorer
*   Create `testing/quality/score.ts`.
*   **Logic:**
    1. Parse the generated `bundle.js.map` using the `source-map` library and extract the `names` array (the ground truth).
    2. Parse the JS Cartographer output using Babel (`@babel/parser`).
    3. Traverse the AST and collect all `Identifier` names.
    4. Calculate the intersection of the two sets: `(Matching Names / Total Original Names) * 100`.
    5. Output the score to the console.

### 4. Implement Snapshot Differ
*   Create `testing/quality/snapshot.ts`.
*   **Logic:**
    1. Read the previous "golden" snapshot from `testing/quality/snapshots/`.
    2. Read the new output from JS Cartographer.
    3. If they match, report success.
    4. If they differ, print a colored diff to the console.
    5. Prompt the developer: "Does this diff look better or worse? (y/n)". If accepted, overwrite the golden snapshot.

### 5. Create the Quality Runner
*   Create `testing/quality/runner.ts` to orchestrate both tools.
*   Add a script to the main `package.json`: `"test:quality": "tsx testing/quality/runner.ts"`.

## Verification
*   Run `npm run test:quality`.
*   Verify the Source Map Recovery Score outputs a valid percentage.
*   Verify the diffing tool detects changes, prompts the user, and correctly updates the snapshot upon acceptance.
*   Ensure running `npm run test` remains fast and unaffected by this new suite.
